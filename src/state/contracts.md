1) Создание хранилища по начальным данным
Должен быть способ создать внутреннюю «коробку», в которой лежат данные.
Ты передаёшь стартовое значение — менеджер сохраняет его у себя.

2) Чтение состояния хранилища
Должен быть простой способ получить текущие данные из хранилища в любой момент.

3) Обновление состояния хранилища
Должен быть один чёткий способ изменять данные — напрямую или через функцию.
После обновления менеджер фиксирует новое значение и запускает уведомления.

4) Оповещение об изменении данных
Менеджер должен отправлять сигнал, что данные поменялись.
Сигнал должен приходить всем, кто подписался.

5) Подписки на изменение
Да. Должен быть способ зарегистрировать функцию, которая будет вызвана при изменении данных.

6) Возможность отписки
Должен быть простой способ отключить обработчик, чтобы его больше не вызывали.

7) Ограничения:
Минимальный набор правил, чтобы хранилище было предсказуемым:

Обновления происходят последовательно
Подписчики вызываются после обновления
Нельзя читать или менять внутренние данные, минуя публичный интерфейс
Подписчик не должен ломать хранилище (например, кидать ошибку наружу)
Изменение должно быть явным — случайные мутированные объекты недопустимы
Нельзя уведомлять подписчиков, если значение не изменилось (опционально)

Данные меняются только через официальный путь, а поведение при обновлении строго определено.

Создал хранилище -> Подписался -> Обновил данные -> Подписчик получил уведомление
Создал хранилище -> Подписался -> Обновил данные -> Подписчик получил уведомление -> Отписался
Создал хранилище -> Получил данные
